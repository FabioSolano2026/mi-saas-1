🏗️ Resumen Ejecutivo

Sistema multi-tenant, multi-nicho con un núcleo CRM universal al que se conectan módulos especializados por industria. El agente conversacional califica prospectos, recomienda productos con lenguaje legal exacto, agenda citas si el nicho lo requiere, y deposita la tarjeta en el Kanban del socio.

Primer nicho validado: MLM Bienestar — validación de ingredientes, cumplimiento regulatorio por país, y ciclos de recompra de 6 meses.

🔑
Regla de oro: Frontend NUNCA llama a Supabase directamente

Todo flujo pasa por API Routes. El servidor verifica auth.getUser() y aplica RLS. El frontend no envía tenant_id ni socio_id — el servidor los toma del JWT.

⚡ Funciones PostgreSQL
set_actualizado_en()
Trigger compartido. Actualiza actualizado_en = now() en cualquier tabla que lo tenga. Verificado en test 4 de auditoría.
mover_kanban(p_prospecto_id, p_socio_id, p_destino, p_nota)
Transacción atómica: 1) UPDATE prospectos.columna_kanban 2) INSERT movimientos_kanban. Esquema: public. Llamada vía supabase.rpc('mover_kanban', ...).
agendar_cita(...)
Reserva slot con FOR UPDATE NOWAIT. Doble-booking imposible bajo concurrencia. Falla inmediatamente si el slot está siendo bloqueado por otro proceso.
custom_access_token_hook(event)
Inyecta tenant_id en el JWT al hacer login. Vive en schema public. Debe estar activado en Dashboard → Auth → Hooks → Customize Access Token.
handle_new_user()
Trigger en auth.users. Al registrar: crea tenant + socio automáticamente. No necesita intervención manual.
🚨
Auth Hook — Configuración manual requerida

El hook custom_access_token_hook debe estar activado en Supabase Dashboard → Authentication → Hooks → "Customize Access Token (JWT) Claims". Sin él, todas las políticas RLS fallan (tenant_id no llega al JWT).

🔒 RLS & Seguridad

Patrón universal de aislamiento multi-tenant

-- SELECT: cada socio ve solo su tenant
USING (tenant_id = (auth.jwt() ->> 'tenant_id')::uuid)

-- INSERT: el socio solo crea en su tenant y como su propio usuario
WITH CHECK (
  tenant_id = (auth.jwt() ->> 'tenant_id')::uuid
  AND socio_id = auth.uid()
)

Kanban inmutable — movimientos_kanban

-- Solo INSERT está permitido
CREATE POLICY movimientos_insert ON movimientos_kanban
  FOR INSERT WITH CHECK (socio_id = auth.uid());
-- Sin política UPDATE → bloqueado automáticamente
-- Sin política DELETE → bloqueado automáticamente

🐛 Errores Críticos Resueltos (Auto-blindaje)

1. Auth Hook en schema incorrecto

Problema: Management API bloqueó crear custom_access_token_hook en schema auth.
Fix: Crearlo en schema public. Supabase Dashboard acepta hooks de cualquier schema.

2. createServerClient devuelve tipo never

Problema: El overload deprecado options: any era seleccionado, devolviendo SupabaseClient<Database, ..., never>. Todos los .insert() tipaban como never.
Fix: Importar CookieMethodsServer para forzar el overload correcto + cast as unknown as SupabaseClient<Database>.

3. Bug crítico — columnas Kanban no coincidían con BD

Problema: Código tenía calificado, interesado, negociacion, perdido, descartado. La BD tiene 9 valores completamente diferentes.
Fix: Reemplazados en kanban.types.ts, KanbanColumn.tsx, MoverModal.tsx, kanban.service.ts. Detectado por audit test antes del merge.

4. FK violation en audit test

Problema: Test usaba UUID 00000000-...-0000 como p_socio_id → violaba FK constraint.
Fix: Query real del primer socio del tenant demo antes de llamar la función.
